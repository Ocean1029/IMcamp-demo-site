## 1 把網站放到網路上（20 分鐘）

### 1.1 什麼是「部署」？

你的網站目前只存在自己的電腦裡，別人沒有辦法打開它。要讓其他人也能用網址看到你的網站，就需要把檔案放到一台 24 小時開著的電腦（伺服器）上。這個動作叫做**部署 (Deploy)**。

今天我們用 **GitHub Pages** 來部署，它是 GitHub 內建的免費服務，設定最簡單。

### 1.2 GitHub Pages 部署實作

請跟著以下步驟操作：

1. 確認你在網頁設計（一）的作品已經推上 GitHub repository。

2. 前往你的 repository 頁面，點擊上方的 **Settings**

<!-- TODO: 截圖 — repository 頁面，標注 Settings 按鈕位置 -->

3. 在左側選單找到 **Pages**

<!-- TODO: 截圖 — Settings 左側選單，標注 Pages 位置 -->

4. 在 **Source** 區塊：
- 選擇 **Deploy from a branch**
- **Branch** 選 `main`，資料夾選 `/ (root)`
- 按下 **Save**

<!-- TODO: 截圖 — Pages 設定頁面，標注 Source、Branch、Save 的位置 -->

5. 等待 1-2 分鐘，頁面上方會出現你的網址：
```
https://<你的 GitHub 帳號>.github.io/<repository 名稱>/
```

<!-- TODO: 截圖 — 部署成功後顯示網址的畫面 -->

6. 部署成功，用電腦直接輸入網址，確認網站有正常顯示。

> **📋 助教提示：** 確認每個人都成功拿到網址，協助處理部署失敗的同學。

### 1.3 GitHub Pages 的自動部署

設定完成後，GitHub Pages 會自動監聽你的 repository。也就是說：

```
你在本地改程式碼 → git add → git commit → git push → GitHub Pages 自動重新部署
```

不用再手動設定，每次 push 到 GitHub，GitHub Pages 都會幫你自動更新網站。

### 1.4 部署常見問題

如果部署後網站沒有出現或顯示異常，到 repository 的 **Actions** 頁籤可以看到部署狀態。

<!-- TODO: 截圖 — Actions 頁籤，標注部署成功/失敗的狀態 -->

最常見的原因：

| 錯誤 | 原因 | 解法 |
|------|------|------|
| 404 Not Found | 檔案名稱不對 | 確認首頁檔名是 `index.html`（不是 `Index.html` 或其他） |
| 畫面空白 | CSS/JS 路徑錯誤 | 確認用相對路徑（`./style.css`）而非絕對路徑 |
| 部署很久沒更新 | GitHub Actions 還在跑 | 到 Actions 頁籤確認狀態，通常等 1-2 分鐘即可 |

## 2 當 AI 寫壞了怎麼辦（10 分鐘）

用 Cursor 寫程式很快，但 AI 不是每次都寫對，它也會寫出有 Bug 的程式碼。所以你得學會自己看錯誤訊息，才能判斷 AI 到底有沒有修好。

### 2.1 兩種錯誤，兩個地方看

開發網站時，錯誤訊息會出現在兩個地方，各代表不同的問題：

**終端機錯誤 (Terminal Error)**：在 Cursor 下方的 Terminal 看到

<!-- TODO: 截圖 — Cursor Terminal 顯示紅色錯誤訊息的畫面 -->
- 代表你的程式碼根本跑不起來
- 常見原因：少裝套件、語法寫錯、檔案路徑打錯

```
# 範例：找不到元件
Module not found: Can't resolve './Compnent'
# 原因：Component 拼成了 Compnent（少一個 o）
```

**瀏覽器 Console 錯誤**：用 F12（或 Cmd+Opt+I）打開開發者工具 → Console 頁籤

<!-- TODO: 截圖 — 瀏覽器 Console 顯示紅色錯誤的畫面 -->
- 網站有跑起來，但某個功能壞了
- 常見原因：變數是 undefined、API 回傳錯誤、找不到某個元素

```
# 範例：存取不存在的屬性
Uncaught TypeError: Cannot read properties of undefined (reading 'name')
# 原因：資料還沒載入就去讀它的 name
```

> **📋 助教提示：** 可以現場示範，故意拼錯 component 名稱，帶大家看錯誤出現在哪裡。

打開任意一個網站，按 F12 → Console，看看有沒有黃色警告或紅色錯誤。連大公司的網站都不一定是零錯誤的。

### 2.2 Debug 三步驟

遇到錯誤時，按照這三步處理：

1. **讀紅字**：錯誤訊息的最後幾行通常最重要，它會告訴你是哪個檔案的哪一行出了問題。
2. **貼給 AI**：把錯誤訊息複製起來，貼到 Cursor 的 Chat，讓 AI 幫你分析。
3. **檢查 AI 的修改**：不要盲目按 Accept，看看 AI 改了什麼、為什麼這樣改。用 Cursor 的 diff view 確認。

<!-- TODO: 截圖 — Cursor 的 diff view，標注 Accept / Reject 按鈕 -->

> **重要提醒：** 如果 AI 越修越壞（改了 A 壞了 B，改了 B 又壞了 C），直接按 `Ctrl+Z`（Mac: `Cmd+Z`）復原，或用 git 回到上一個正常的版本。不要讓 AI 無止盡地改下去。

## 3 用別人的服務讓網站更有趣（10 分鐘）

### 3.1 什麼是 API？

如果想在網站上顯示天氣、地圖、隨機圖片這些資料，不用自己從零寫起，可以直接跟別人的伺服器「要資料」。這個要資料的方式就叫 **API (Application Programming Interface)**。

具體來說就是：你的網站透過一個 URL 發送請求（Request），對方的伺服器收到後回傳資料（Response）。這些別人寫好、開放給你用的 API 就叫**第三方 API**，不用自己寫後端，網站就能有這些功能。

### 3.2 用瀏覽器看 API 回傳的資料

在瀏覽器網址列直接輸入以下 URL，看看回傳什麼：

```
https://dog.ceo/api/breeds/image/random
```

你會看到類似這樣的 JSON 格式資料：
```json
{
  "message": "https://images.dog.ceo/breeds/terrier-norfolk/n02094114_1842.jpg",
  "status": "success"
}
```

你用網址發請求，伺服器就回資料給你，就這樣。

### 3.3 實作：在你的網站串接 API

接下來用 Cursor 串接 Dog CEO API，在網站上顯示隨機狗狗圖片。

**Prompt 範例：**
```
請幫我在網站上新增一個「隨機狗狗」區塊：
1. 頁面載入時，從 https://dog.ceo/api/breeds/image/random 取得一張隨機狗狗圖片並顯示
2. 加一個「換一張」按鈕，點擊後重新取得並顯示新的圖片
3. 圖片載入時顯示 loading 狀態
4. 風格要跟網站其他部分一致
```

你也可以試試這些免費 API（不需要 API Key）：
- **隨機名言：** `https://zenquotes.io/api/random`
- **隨機笑話：** `https://official-joke-api.appspot.com/random_joke`
- **假資料產生器：** `https://jsonplaceholder.typicode.com/posts`

### 3.4 API Key 與資安

有些 API（像是 Google Maps、OpenAI）需要 **API Key** 才能用。API Key 等同密碼，**絕對不要寫在程式碼裡推上 GitHub**，不然別人拿去亂用，帳單會寄到你頭上。今天用的 API 都不需要 Key，未來如果用到，再查 `.env` 和 `.gitignore` 的用法就好。

## 4 響應式網頁設計 RWD（10 分鐘）

### 4.1 什麼是 RWD？

用手機打開某些網站，文字超小、排版亂掉、要一直左右滑，那就是沒有處理手機版的畫面。

**RWD（Responsive Web Design，響應式網頁設計）** 的意思就是：同一份程式碼，螢幕寬度不同，排版就會自動調整。

```
桌面版 (>1024px)          平板 (768-1024px)         手機版 (<768px)
┌──────────────────┐     ┌──────────────┐          ┌────────┐
│  Logo    Nav Nav │     │  Logo   ≡    │          │ Logo ≡ │
├──────────────────┤     ├──────────────┤          ├────────┤
│ Card  Card  Card │     │ Card   Card  │          │  Card  │
│ Card  Card  Card │     │ Card   Card  │          │  Card  │
└──────────────────┘     └──────────────┘          │  Card  │
                                                   └────────┘
```

### 4.2 實驗：用 F12 模擬手機畫面

1. 打開你部署好的網站
2. 按 F12 開啟開發者工具
3. 開發者工具打開後，點左上角的「裝置模擬」按鈕（兩個重疊的矩形 icon），或按 `Ctrl+Shift+M`（Mac: `Cmd+Shift+M`）

<!-- TODO: 截圖 — 開發者工具，標注裝置模擬按鈕位置 -->

4. 上方可以選不同的裝置型號（iPhone、iPad 等），或自訂寬度

<!-- TODO: 截圖 — 裝置模擬模式下的網站畫面，標注裝置選擇下拉選單 -->

看看你的網站在手機上長什麼樣子。如果跑版了，接下來我們會處理。

### 4.3 用 Cursor 實作 RWD

這裡有幾個 prompt，可以直接貼進 Cursor：

**漢堡選單：**
```
請幫我把導覽列在手機版（螢幕寬度小於 768px）改成漢堡選單：
- 導覽列只顯示 Logo 和一個 ≡ 按鈕
- 點擊按鈕後展開完整的選單
- 選單展開時有平滑的動畫效果
```

**單欄排版：**
```
請讓卡片區塊在手機版改為單欄排列，每張卡片佔滿整個寬度
```

**字體調整：**
```
請調整手機版的字體大小：
- 大標題從 48px 縮小到 28px
- 副標題從 24px 縮小到 18px
- 內文保持 16px（手機上 16px 是最佳閱讀大小）
```

改完之後，記得用 F12 模擬手機畫面確認效果。也可以 push 到 GitHub，等 GitHub Pages 部署完，直接用手機打開看。

## 5 自由實作（15 分鐘）

### 5.1 Prompt 靈感

可以加的功能：

**動畫特效：**
```
請幫我加入打字機特效，讓首頁的標題文字一個字一個字出現
```

**滑入動畫：**
```
請在每個區塊加入滑入動畫，當使用者滾動到該區塊時才從下方滑入出現
```

**深色模式：**
```
請加一個深色模式切換按鈕，點擊後整個網站切換成深色主題，
並且把使用者的偏好儲存在 localStorage 裡面
```

**回到頂部按鈕：**
```
請加一個「回到頂部」的浮動按鈕，只在使用者往下滾動後才出現，
點擊後平滑滾動到頁面最上方
```

**頁尾設計：**
```
請幫我設計一個好看的 footer，包含社群連結 icon
```

**訪客計數器（進階，需要外部 API）：**
```
請串接一個免費的訪客計數 API，在頁尾顯示這個網站被瀏覽了幾次
```

### 5.2 實作小提醒

- **一次改一個功能：** 不要同時叫 AI 改三四樣東西，容易出問題
- **每完成一個功能就 commit：** `git add index.html style.css script.js` → `git commit -m "加入深色模式"` → `git push`（避免用 `git add .`，以免不小心推上 `.env` 等敏感檔案）
- **善用 Debug 流程：** 遇到錯誤就用剛剛學的三步驟
- **記得看手機版：** 加完功能後用 F12 或手機確認 RWD 有沒有壞掉

> **📋 助教提示：** 巡場確認大家的部署都正常運作，協助處理 Debug 問題，鼓勵學生嘗試不同的 prompt 寫法。
