## 1 讓全世界看到你的網站（20 分鐘）

### 1.1 什麼是「部署」？

在網頁設計（一）中，你的網站一直跑在 `localhost`（本機），只有你自己的電腦看得到。如果你想讓朋友、家人、甚至全世界的人都能看到你的網站，就需要把它放到一台 24 小時運作的伺服器上——這個過程就叫做**部署 (Deploy)**。

用餐廳的比喻來說：

| 階段 | 比喻 | 對應 |
|------|------|------|
| 本地開發 | 在家裡廚房練習做菜 | 在自己電腦上用瀏覽器打開 |
| 推上 GitHub | 把食譜寫好存在雲端 | `git push` |
| **部署** | **把餐廳開起來，讓客人進來吃** | **GitHub Pages** |

常見的部署平台：
- **GitHub Pages** — GitHub 內建的免費靜態託管，設定簡單（今天用這個）
- **Vercel** — 與 GitHub 整合好，對 Next.js / React 專案特別友善
- **Netlify** — 也很好用，適合靜態網站

### 1.2 GitHub Pages 部署實作

請跟著以下步驟操作：

**(1)** 確認你在網頁設計（一）的作品已經推上 GitHub repository。

**(2)** 前往你的 repository 頁面，點擊上方的 **Settings**

**(3)** 在左側選單找到 **Pages**

**(4)** 在 **Source** 區塊：
- 選擇 **Deploy from a branch**
- **Branch** 選 `main`，資料夾選 `/ (root)`
- 按下 **Save**

**(5)** 等待 1-2 分鐘，頁面上方會出現你的網址：
```
https://<你的 GitHub 帳號>.github.io/<repository 名稱>/
```

**(6)** 部署成功！用手機直接輸入網址，看看你的網站在手機上長什麼樣子！

> **📋 助教提示：** 確認每個人都成功拿到網址，協助處理部署失敗的同學。

### 1.3 GitHub Pages 的自動部署

設定完成後，GitHub Pages 會自動監聽你的 repository。也就是說：

```
你在本地改程式碼 → git add → git commit → git push → GitHub Pages 自動重新部署
```

你不需要再手動設定，每次 push 到 GitHub，GitHub Pages 都會幫你自動更新網站。這就是業界常說的 **CI/CD (持續整合/持續部署)** 的基本概念。

### 1.4 部署常見問題

如果部署後網站沒有出現或顯示異常，不要慌！到 repository 的 **Actions** 頁籤可以看到部署狀態。最常見的原因：

| 錯誤 | 原因 | 解法 |
|------|------|------|
| 404 Not Found | 檔案名稱不對 | 確認首頁檔名是 `index.html`（不是 `Index.html` 或其他） |
| 畫面空白 | CSS/JS 路徑錯誤 | 確認用相對路徑（`./style.css`）而非絕對路徑 |
| 部署很久沒更新 | GitHub Actions 還在跑 | 到 Actions 頁籤確認狀態，通常等 1-2 分鐘即可 |

## 2 當 AI 寫壞了怎麼辦（10 分鐘）

在網頁設計（一）中，你已經體驗了用 Cursor 寫程式的效率。但 AI 不是萬能的，它也會寫出有 Bug 的程式碼。學會看懂錯誤訊息、找出問題，是成為開發者最重要的技能之一。

### 2.1 兩種錯誤，兩個地方看

開發網站時，錯誤訊息會出現在兩個不同的地方，代表不同類型的問題：

**終端機錯誤 (Terminal Error)** — 在 Cursor 下方的 Terminal 看到
- 這類錯誤通常發生在**編譯階段**，代表你的程式碼根本跑不起來
- 常見：少了套件、語法寫錯、檔案路徑不對

```
# 範例：找不到元件
Module not found: Can't resolve './Compnent'
# 原因：Component 拼成了 Compnent（少一個 o）
```

**瀏覽器 Console 錯誤** — 用 F12（或 Cmd+Opt+I）打開開發者工具 → Console 頁籤
- 這類錯誤發生在**執行階段**，網站跑起來了但某些功能壞了
- 常見：變數是 undefined、API 回傳錯誤、元素找不到

```
# 範例：存取不存在的屬性
Uncaught TypeError: Cannot read properties of undefined (reading 'name')
# 原因：資料還沒載入就試著去讀取它的 name
```

> **📋 助教提示：** 可以現場示範，故意拼錯 component 名稱，帶大家看錯誤出現在哪裡。

**實驗：** 打開任意一個網站（例如你自己的），按 F12 → Console，看看有沒有任何黃色警告或紅色錯誤。你會發現，連大公司的網站都不一定是零錯誤的！

### 2.2 Debug 三步驟

當你遇到錯誤時，不要害怕，按照這三步走：

```
Step 1 ─── 讀紅字
            錯誤訊息的「最後幾行」通常最重要，它會告訴你是哪個檔案的哪一行出了問題。
            ↓
Step 2 ─── 貼給 AI
            把錯誤訊息複製起來，貼到 Cursor 的 Chat 或 Debug 模式，
            讓 AI 幫你分析。
            ↓
Step 3 ─── 檢查 AI 的修改
            不要盲目按 Accept！看看 AI 改了什麼，試著理解它的修正邏輯。
            用 git diff 或 Cursor 的 diff view 來確認。
```

> **重要提醒：** 如果 AI 越修越壞（改了 A 壞了 B，改了 B 又壞了 C），請果斷使用 `Ctrl+Z`（Mac: `Cmd+Z`）復原，或者用 git 回到上一個正常的版本。不要讓 AI 無止盡地改下去。

## 3 第三方 API 串接（10 分鐘）

### 3.1 什麼是 API？

在網頁設計（一）中有提到前端和後端的概念。**API (Application Programming Interface)** 就是前端和後端溝通的橋樑。

沿用餐廳的比喻：

| 角色 | 對應 | 說明 |
|------|------|------|
| 客人（你） | 前端 | 想要取得某些資料 |
| 菜單 | API 文件 | 告訴你有哪些東西可以點、要怎麼點 |
| 服務生 | HTTP 請求 | 負責把你的需求送到廚房 |
| 廚房 | 後端伺服器 | 準備好資料 |
| 上菜 | API 回應 (Response) | 把資料送回給你 |

**第三方 API** 就是別人寫好、開放給你用的 API。例如 Google Maps、天氣資料、隨機圖片等等。你不需要自己寫後端，就能讓網站擁有這些功能。

### 3.2 實驗室：用瀏覽器看 API 回傳的資料

在瀏覽器網址列直接輸入以下 URL，看看回傳什麼：

```
https://dog.ceo/api/breeds/image/random
```

你會看到類似這樣的 JSON 格式資料：
```json
{
  "message": "https://images.dog.ceo/breeds/terrier-norfolk/n02094114_1842.jpg",
  "status": "success"
}
```

這就是 API 的運作方式——你發送一個請求 (Request)，伺服器回傳一個回應 (Response)。

### 3.3 實作：在你的網站串接 API

現在讓我們用 Cursor 來串接這個 Dog CEO API，在你的網站上顯示隨機狗狗圖片。

**Prompt 範例：**
```
請幫我在網站上新增一個「隨機狗狗」區塊：
1. 頁面載入時，從 https://dog.ceo/api/breeds/image/random 取得一張隨機狗狗圖片並顯示
2. 加一個「換一張」按鈕，點擊後重新取得並顯示新的圖片
3. 圖片載入時顯示 loading 狀態
4. 風格要跟網站其他部分一致
```

其他你可以嘗試串接的免費 API（不需要 API Key）：
- **隨機名言：** `https://zenquotes.io/api/random`
- **隨機笑話：** `https://official-joke-api.appspot.com/random_joke`
- **假資料產生器：** `https://jsonplaceholder.typicode.com/posts`

### 3.4 API Key 與資安

有些比較強大的 API（像是 Google Maps、OpenAI）需要一個 **API Key** 才能使用。API Key 就像是你的「通行證」，用來辨識是誰在呼叫這個 API。

> **絕對不要把 API Key 直接寫在程式碼裡面！**

如果你把含有 API Key 的程式碼推到 GitHub，全世界的人都看得到。有人可能會拿你的 Key 去亂用，帳單就會寄到你頭上。

**安全的做法：**
1. 把 API Key 存在 `.env` 檔案裡（環境變數）
2. 確認 `.gitignore` 有包含 `.env`，這樣 git 不會把它推上去

```bash
# .env 檔案（這個檔案不會被推上 GitHub）
MAP_API_KEY=你的_API_KEY_放這裡
```

```bash
# .gitignore 檔案（確認裡面有這一行）
.env
```

> **注意：** `.env` 需要搭配打包工具（如 Vite、Next.js）才能使用。如果你是寫純 HTML/CSS/JS 的靜態網站，目前不會用到需要 API Key 的服務，這段先了解概念即可。

> 如果你不小心已經把 Key 推上去了：光是刪除那個 commit 是不夠的（git 歷史還是看得到）。正確做法是**立刻去 API 提供者的後台重新產生一組新的 Key**，舊的就自動失效了。

## 4 響應式網頁設計 RWD（10 分鐘）

### 4.1 什麼是 RWD？

你有沒有用手機打開過一些網站，結果文字超小、排版亂七八糟、要一直左右滑才看得到內容？那就是沒有做**響應式網頁設計 (Responsive Web Design, RWD)** 的結果。

RWD 的核心概念：**同一份程式碼，根據螢幕寬度自動調整排版。**

```
桌面版 (>1024px)          平板 (768-1024px)         手機版 (<768px)
┌──────────────────┐     ┌──────────────┐          ┌────────┐
│  Logo    Nav Nav │     │  Logo   ≡    │          │ Logo ≡ │
├──────────────────┤     ├──────────────┤          ├────────┤
│ Card  Card  Card │     │ Card   Card  │          │  Card  │
│ Card  Card  Card │     │ Card   Card  │          │  Card  │
└──────────────────┘     └──────────────┘          │  Card  │
                                                   └────────┘
```

### 4.2 實驗：用 F12 模擬手機畫面

1. 打開你部署好的網站
2. 按 F12 開啟開發者工具
3. 在開發者工具已開啟的狀態下，點擊左上角的「裝置模擬」按鈕（兩個重疊的矩形 icon），或按 `Ctrl+Shift+M`（Mac: `Cmd+Shift+M`）
4. 上方可以選不同的裝置型號（iPhone、iPad 等），或自訂寬度

看看你的網站在手機上長什麼樣子——如果跑版了，別擔心，接下來我們就來修它！

### 4.3 用 Cursor 實作 RWD

以下是幾個你可以直接貼進 Cursor 的 prompt：

**漢堡選單：**
```
請幫我把導覽列在手機版（螢幕寬度小於 768px）改成漢堡選單：
- 導覽列只顯示 Logo 和一個 ≡ 按鈕
- 點擊按鈕後展開完整的選單
- 選單展開時有平滑的動畫效果
```

**單欄排版：**
```
請讓卡片區塊在手機版改為單欄排列，每張卡片佔滿整個寬度
```

**字體調整：**
```
請調整手機版的字體大小：
- 大標題從 48px 縮小到 28px
- 副標題從 24px 縮小到 18px
- 內文保持 16px（手機上 16px 是最佳閱讀大小）
```

改完之後，記得用 F12 模擬手機畫面確認效果。更好的做法是 push 到 GitHub，等 GitHub Pages 部署完，直接用你的手機打開看！

## 5 Happy Vibe Coding！（15 分鐘）

現在你已經學會了部署、Debug、API 串接、RWD，接下來是屬於你的自由實作時間！

### 5.1 Prompt 靈感

不知道要加什麼功能？這裡有一些點子：

**動畫特效：**
```
請幫我加入打字機特效，讓首頁的標題文字一個字一個字出現
```

**滑入動畫：**
```
請在每個區塊加入滑入動畫，當使用者滾動到該區塊時才從下方滑入出現
```

**深色模式：**
```
請加一個深色模式切換按鈕，點擊後整個網站切換成深色主題，
並且把使用者的偏好儲存在 localStorage 裡面
```

**回到頂部按鈕：**
```
請加一個「回到頂部」的浮動按鈕，只在使用者往下滾動後才出現，
點擊後平滑滾動到頁面最上方
```

**頁尾設計：**
```
請幫我設計一個好看的 footer，包含社群連結 icon
```

**訪客計數器（進階，需要外部 API）：**
```
請串接一個免費的訪客計數 API，在頁尾顯示這個網站被瀏覽了幾次
```

### 5.2 實作小提醒

- **一次改一個功能：** 不要同時叫 AI 改三四樣東西，容易出問題
- **每完成一個功能就 commit：** `git add index.html style.css script.js` → `git commit -m "加入深色模式"` → `git push`（避免用 `git add .`，以免不小心推上 `.env` 等敏感檔案）
- **善用 Debug 流程：** 遇到錯誤就用剛剛學的三步驟
- **記得看手機版：** 加完功能後用 F12 或手機確認 RWD 有沒有壞掉

> **📋 助教提示：** 巡場確認大家的部署都正常運作，協助處理 Debug 問題，鼓勵學生嘗試不同的 prompt 寫法。
